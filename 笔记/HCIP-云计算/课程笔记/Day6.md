# Day6
## 下午

### 一、基础概念
#### 线程、进程与程序的关系

- 程序：存储在磁盘上的静态代码文件

- 进程：程序运行时的 “运行态”，一个程序运行时至少会启动一个进程

- 线程：进程内的执行单元，一个进程需要一个线程来获取 CPU 资源并执行计算

#### CPU 核心与线程的基础关系

- 一般情况下，**单个 CPU 核心只有 1 个线程**，同一时刻最多只能为一个程序（线程）提供计算资源。

- 例如：物理 CPU 为 16 核心，无超线程时，可同时处理 16 个线程的计算任务。

### 二、超线程技术（Hyper-Threading）

定义：让**单个 CPU 核心除了本身的线程外，额外虚拟出 1 个线程**，使一个核心在同一时刻可以处理两个线程的任务，从而提升 CPU 的利用率。

示例：

- 8 核物理 CPU，开启超线程后，可提供 **16 个逻辑线程**（8×2）。
- 对应笔记里的 `8-16` 标注，代表 8 个物理核心通过超线程扩展为 16 个逻辑线程。

### 三、补充说明

- 核心是物理硬件单元，线程是逻辑执行单元。
- 超线程技术能提升多任务场景下的性能，但不能完全等同于翻倍的物理核心性能。

### 四、分时复用技术（CPU 时间片轮转）

分时复用技术：操作系统通过**时间片轮转**，让 CPU 在极短的时间内，高频切换执行不同程序 / 进程的任务。

由于切换速度极快（毫秒级甚至更快），用户在感知上，多个程序是 “同时运行” 的。

本质：**同一物理 CPU 核心，在微观上是串行执行的，宏观上表现为并行**。

这也是虚拟化 CPU 超配（vCPU > 物理 CPU）的核心基础。

> [!NOTE] 单个虚拟机的 vCPU 可以超过物理 CPU 核心数吗
> 1）可以，按照软件来
> 
> VMware Workstation Pro最多 **32 个 vCPU**
> VMware ESXi（企业级）最高支持到 **128 个 vCPU**（受主机和版本限制）
> VirtualBox最多 **32 个 vCPU**
> 
> 你的 CPU 同一时间最多只能执行 8 条线程指令。给虚拟机配置超过 8 个 vCPU，多余的 vCPU 在宿主机层面会处于等待调度的状态，**无法同时执行任何指令**
> 
> 虚拟机内的操作系统会认为自己有很多个 CPU，会发起大量的任务调度、中断处理，这些额外的开销会**增加宿主机的负担**，导致虚拟机的实际运行速度，比配置等于物理线程数时还要慢。
> 
> 不管你配置 16 个、32 个还是更多 vCPU，虚拟机的实际处理能力，**上限永远等于你的物理 CPU 的性能**，不会有任何 “多核加成”。
> 

> [!NOTE] 多个虚拟机的 vCPU 总和可以超过物理 CPU 核心数吗
> 2）可以，CPU超分
> 
> 基于时间片复用，是把物理 CPU 的时间片动态分配给虚拟机：
> 
> 比如你的物理机是 **8 线程**，同时开 3 台虚拟机，每台给 4 个 vCPU（总 vCPU=12），当只有 1 台虚拟机在跑满负载时，它可以独占所有物理线程，速度几乎不受影响；另外两台空闲的虚拟机，只会占用少量调度开销。
> 
> 负载稀疏性：绝大多数虚拟机的 CPU 都不是**持续 100% 满载**的：
> - 办公、测试、挂机类虚拟机，90% 以上的时间都处于低负载 / 空闲状态，只有偶尔会短暂跑满。
> - 虚拟化平台利用这一点，把 “空闲的 CPU 时间” 重新分配给正在忙的虚拟机，从而提升整体资源利用率。
> 
> 举个例子：8 线程物理机，开 4 台虚拟机各给 4 个 vCPU（总 vCPU=16，超配 2 倍），只要同一时间不超过 2 台虚拟机同时满载，整体体验依然流畅。
> 
> 有代价的，多个虚拟机同时进入高负载时，就会出现明显问题
> 
> - **资源争抢与性能骤降**
>     
>     当多个虚拟机同时跑满 CPU 时，物理线程需要在不同虚拟机之间频繁切换调度，上下文切换开销剧增，所有虚拟机都会出现卡顿、响应变慢，甚至比 “不超配时的单台虚拟机” 还慢。
> - **调度延迟与稳定性下降**
>     
>     对于数据库、实时业务这类对延迟敏感的虚拟机，超配可能导致任务调度延迟，引发业务超时、系统不稳定。
> - **没有额外性能上限**
>     
>     无论超配多少，所有虚拟机的总性能**永远不会超过物理 CPU 本身的性能**，超配只是 “分时复用”，不是 “凭空提升算力”。
> 
> 场景类型推荐超配比例原因
> 
> 生产环境 / 高负载业务（数据库、编译、渲染）**≤1:1（不超配）**避免任何资源争抢，保证业务稳定性和性能
> 
> 测试 / 开发 / 办公虚拟机**1:1 ~ 1:2**大部分时间低负载，少量超配可提升资源利用率
> 
> 挂机 / 低负载后台服务**1:2 ~ 1:4**几乎不跑满 CPU，超配收益高，风险低

### 五、CPU Ring 权限等级（笔记核心）

CPU 把指令执行权限分成了 4 个环（Ring 0~Ring 3），权限从内到外依次降低：

|    权限环     | 权限级别 |       典型运行内容       |             关键特性              |
| :--------: | :--: | :----------------: | :---------------------------: |
| **Ring 0** | 最高权限 | 操作系统内核（Host/Guest） | 可以直接执行所有 CPU 敏感指令（如修改页表、开关中断） |
|   Ring 1   | 次高权限 |   操作系统服务 / 外围工具    |       早期系统用，现代操作系统已很少使用       |
|   Ring 2   | 中权限  |    部分系统服务 / 驱动     |           现代系统几乎不使用           |
| **Ring 3** | 最低权限 |   用户应用程序 / 普通进程    |   无法直接执行敏感指令，必须通过系统调用向内核申请    |

### 六、虚拟化原理分类（结合你的笔记修正 & 补充）

#### 1. 半虚拟化（Para-virtualization）

- **原理**：修改虚拟机操作系统内核，让它主动配合虚拟化。Guest 内核不再直接执行敏感指令，而是主动向宿主机 Hypervisor 发出请求。
- **特点**：
    
    - 性能损耗小，不需要捕获 / 模拟指令
    - 缺点：必须修改 Guest OS 内核，**仅开源系统（如 Linux）支持**，Windows 等闭源系统无法使用
    - 典型技术：早期的 Xen（半虚拟化模式）
    

#### 2. 全虚拟化（Full virtualization）

- **原理**：不修改 Guest OS 内核，Guest OS 以为自己运行在真实硬件上。Hypervisor 需要**动态监控并捕获 Guest 发出的所有敏感指令**，然后模拟执行这些指令。
- **特点**：
    
    - 优点：Guest OS 无需修改，支持 Windows、Linux 等所有系统
    - 缺点：指令捕获 + 模拟开销大，**性能损耗高**
    - 典型技术：早期的 VMware Workstation、QEMU 纯软件模拟
    

#### 3. 硬件辅助虚拟化（Hardware-assisted virtualization）

- **原理**：CPU 厂商（Intel VT-x/AMD-V）修改硬件架构，新增了专门的虚拟化扩展指令集和模式（如 VMX），让 Hypervisor 可以直接在硬件层面处理 Guest 的敏感指令，无需软件模拟。
- **特点**：
    
    - 性能接近物理机，是现在虚拟化的主流方案
    - 无需修改 Guest OS 内核，也不需要软件捕获指令
    - 难度大：需要 CPU 硬件本身支持
    - 典型技术：KVM、VMware ESXi、华为 FusionCompute 的 CNA
    
### 七、华为 FusionCompute 的虚拟化实现

笔记里的内容可以整理为：

> 华为 FusionCompute 的计算节点（CNA），是基于 KVM 二次开发的，采用的是**硬件辅助全虚拟化**技术。

- 说明：
    
    - 基于 Intel VT-x/AMD-V 硬件辅助虚拟化，解决了纯软件全虚拟化的性能损耗问题
    - 同时保留了全虚拟化的优势：支持所有 Guest OS（Windows、Linux 等），无需修改内核
    - 这也是企业级虚拟化平台的标准实现方式

### 八、内存超分配（内存超配）

> 多个虚拟机分配的内存总和可以超过宿主机的物理内存总量，这是虚拟化平台提升内存利用率的核心技术。

它主要通过以下几种机制实现：
#### 1. 内存置换（Swap）

- **原理**：将虚拟机中长时间不访问的内存数据，置换到宿主机磁盘上的专用交换分区（Swap）中；当数据被重新访问时，再从 Swap 交换回物理内存。
- **华为 FusionCompute 细节**：CNA 节点安装时，会默认从系统磁盘中划分出 **30GB** 作为内存交换空间。
- **置换算法**：采用 **LRU（最近最少使用）算法**，优先置换最久未被访问的内存页。
- **⚠️ 注意**：Swap 基于磁盘，速度远低于物理内存，过度使用会导致虚拟机性能严重下降。

#### 2. 内存共享（内存气泡 / 页共享）

- **原理**：对于多个虚拟机中内容完全相同的内存页（如操作系统内核、公共库文件），宿主机只在物理内存中保留一份副本，让所有虚拟机共享读取。
- **写时复制（COW）机制**：当某个虚拟机需要修改共享内存页时，平台会为该虚拟机单独开辟一份新的内存副本，修改操作不会影响其他虚拟机。
- **作用**：大幅减少相同系统镜像的内存占用，提升内存利用率。

#### 3. 内存气泡（Ballooning）

- **原理**：宿主机通过在非关键虚拟机中启动一个 “气泡” 进程，主动回收这些虚拟机的空闲内存，分配给急需内存的关键虚拟机使用。
- **工作方式**：当宿主机内存耗尽或关键虚拟机内存不足时，气泡进程会向虚拟机申请内存，迫使虚拟机释放空闲页，从而 “压缩” 非关键虚拟机的内存占用。
- **优势**：相比 Swap，内存气泡是在内存层面主动回收，性能损耗远低于磁盘置换。
